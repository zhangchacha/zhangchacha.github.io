一个github pages对应多个域名

发表于 2019-09-26 | 评论数： | 阅读次数：

在本博客处于初级阶段的时候，我们的博客的面向对象只有张茶茶和我，因此，一个中文域名完全可以handle。但是，当我决定将它和面试挂上钩之后，它就肩负了不同的使命。

创建一个网站的过程很简单，你甚至什么都不需要，只要在github上新建一个仓库，特别的，这个仓库的名称必须是{username}.github.io，其中username是你的github账户名。如果你的仓库有index.html的话，在{username}.github.io这个域名上显示的就是你的网页了。如果没有的话，会显示你的README中的内容。如果还没有的话，emmmm我也不知道了。

建站工具可以考虑hexo，网上有很多详细的教程，这里提一句，hexo有很多主题，最大名鼎鼎的应该是next主题，支持各种你看得懂或者看不懂的配置。请酌情食用。

假设我已经建好网站，并且部署到了github pages的网页上了。只要有人点击我的github.io网址就会看到我的网站。但是新的需求又来了。我想要一个更加fancy的域名。github.io太长了，而且一看就是依托于github的，不利于传播自己的形象。于是我斥巨资（9块/年）在某处买了一个域名。我们假设是foo.com吧。现在我们想做的事情是把foo.com和我的github pages绑定起来。

域名绑定

绑定过程就不止是我们做的事情了。需要我们的域名提供商为我们做到这件事。首先在域名解析处添加CNAME解析，或者A解析。事实上我并不知道CNAME解析和A解析本质上的区别，只是知道A解析将网页解析到一个IPV4地址，而CNAME解析将网页解析到另一个域名。

CNAME解析到另一个域名可能和直观的想法不同，解析到另一个域名并不是说当你访问这个域名时直接跳转到另一个域名（这种东西当然有，叫url转发，但是有其他限制），具体做了什么我也并不清楚。

于是我们添加了四个A解析和一个CNAME解析。其中A解析的前缀是@，表示直接输入foo.com时候的解析，CNAME解析的前缀是www，表示输入www.foo.com时的解析。四个A解析的地址来自网络，应该很容易搜得到，如果不行的话，直接ping自己的github.io域名也能获取其中一个，一般也就够用了。

这还不算完。我们知道IP是很宝贵的资源，整个github可能都没有多少IP但是你一个人就想独占？不可能的。在你将域名解析到github的IP之后，他会给你做进一步的解析，来确定究竟是那个github pages的访问来了，然后进行处理。

这时候我们需要打开github.io仓库的设置，将custome domain设置为我们的域名。或者，如果你是个命令行爱好者，你可以在自己的仓库的主目录下新建一个CNAME文件（文件名就是这个，没后缀），将自己的网址foo.com填进去。如果你感兴趣的话，设置域名会自动生成一次commit。

然后等待DNS解析完成之后，我们就可以愉快的访问自己的网站了。github pages也会很配合的在浏览器的地址栏显示我们的foo.com。一切都那么安详美好。

多个域名

前面说到，当我决定把这个网站和面试挂上钩的时候，这一切就不那么单纯了。首先，之前我购买的域名不是我自己的名字，其次它是个中文名。不是自己的名字这个问题可能比较大。因为你很可能直接借一个别人的博客来搪塞。至于中文名倒是问题不大，但是我个人觉得可能有一点问题。

于是，我再一次斥巨资（另外9块）买了一个新的域名。这次的域名采用了我的名字的缩写，跟我的qq邮箱名几乎相同。即便是有人会怀疑这个是不是我的网站，至少说明我有一个自己的域名，还是有心想要做一个网站的。

刚买来的时候，我还没有意识到问题的严重性。彼时我用的全是CNAME解析，可以很自然的把自己的github.io域名加到解析目标里，这让人觉得这只是一个简单的链接跳转。

所以去查资料。方法倒是有一些：

  • 使用另一个平台，比如coding。这个似乎是csdn的代码平台，但是我也不确定。hexo或者next支持同时将代码部署到多个平台上，我们相当于维护了两个静态页面仓库。对于正在做去重工作的我，显然是不太能接受的。

  • url转发。我前边提到过，我原本的想法就是让新的网址直接转发到原网址，但是事与愿违，网址转发需要两个网址（转发的原网址和目标网址）都备案。某宝云的备案系统需要有一台主机（似乎看到有人说不用，只要备案过程中网站不可访问就行，但是我没找到方法）。这就很麻烦了。如果能跳过这一步的话，这个方法应该是我的首选。不过没办法，我天朝自有国情在此。

  • 新建跳转仓库。最终使用了这种方法。首先新建一个仓库，可以在任何地方。然后打开github pages服务（大概是选一个branch，前提是你要有branch，很久以前branch的名字是固定的，现在可以随便选了）。将你的新域名绑定到新仓库的custome domain里，然后新域名的地址解析到github.io或者那几个IP。然后要做的事情就是在新的仓库中新建一个index.html文件。如果不放心的话也可以多新建一个404.html。内容主要是跳转到刚才的网页。对完全不懂html的人来说，可以参考这个：

    content标签的参数中，分号之前是跳转前的等待时间，之后是跳转的目标网址。把这个网站弄得好看一点也是可以的，但是我太懒了。

后记

这个东西还是折腾了一段时间的。主要是DNS解析每次更新的延迟大概有十分钟，十分钟之后我可能就不知道去哪了，所以这个事情弄的断断续续的。

等我有钱了就去弄个302转发，谁都别拦着我，有钱了谁还受这个气啊。
